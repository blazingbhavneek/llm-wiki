# 6.2.8.7.8.5 Conditional SWITCH Nodes

Part of [Cuda C Programming Guide Reference](README.md). Source lines L3338-L3810.

- [Conditional SWITCH Nodes](../../../concepts/conditional-switch-nodes.md) — SWITCH nodes, introduced in CUDA 12.8, allow a conditional node to execute one of n different subgraphs based on an evaluated condition value, where the nth graph is selected if the value equals n, and no graph is executed if the value is greater than or equal to n.
- [CUDA Events](../../../concepts/cuda-events.md) — CUDA events provide a mechanism to monitor device progress and perform accurate timing by asynchronously recording checkpoints that complete when all preceding tasks or commands in a specified stream have finished.
- [Synchronous Calls](../../../concepts/synchronous-calls.md) — Synchronous functions block the host thread until the device completes the requested task, with post-completion behavior (yield, block, or spin) configurable via cudaSetDeviceFlags().
- [Multi-Device Systems](../../../concepts/multi-device-systems.md) — Overview of managing multiple CUDA-enabled devices on a host system, including enumeration, selection, and inter-device communication.
- [Device Selection](../../../concepts/device-selection.md) — The device a host thread operates on is set using `cudaSetDevice()`, which determines where memory allocations, kernel launches, streams, and events are associated.
- [Stream and Event Behavior in Multi-Device Systems](../../../concepts/stream-event-behavior-multi-device.md) — Details on how streams and events behave across multiple devices, including failure conditions for kernel launches and event operations, success conditions for memory copies and synchronization, and the independence of default streams.
- [Peer-to-Peer Memory Access](../../../concepts/peer-to-peer-memory-access.md) — Peer-to-Peer (P2P) memory access allows kernels on one GPU to directly dereference pointers to memory allocated on another GPU, provided the system topology supports it and peer access is explicitly enabled.
- [IOMMU on Linux](../../../concepts/iommu-linux.md) — CUDA on Linux bare-metal systems does not support IOMMU-enabled PCIe peer-to-peer memory copies, requiring users to disable IOMMU, whereas virtual machines can use IOMMU with VFIO pass-through.
- [Peer-to-Peer Memory Copy](../../../concepts/peer-to-peer-memory-copy.md) — Peer-to-Peer (P2P) memory copy allows data transfer between different GPU devices using specific CUDA API functions or unified address space, offering synchronization guarantees and performance benefits when peer access is enabled.
- [Unified Virtual Address Space (UVA)](../../../concepts/unified-virtual-address-space.md) — Unified Virtual Address Space (UVA) allows 64-bit applications to use a single address space for host and device memory, enabling portable pointers and simplified memory management.
- [Interprocess Communication (IPC)](../../../concepts/interprocess-communication.md) — Interprocess Communication (IPC) allows sharing device memory pointers and event handles across different processes using the CUDA IPC API, supported on 64-bit Linux for compute capability 2.0+ devices, with specific limitations for managed memory and Tegra platforms.
- [Error Checking](../../../concepts/error-checking.md) — CUDA runtime functions return error codes for host-side validation, while asynchronous errors are managed via thread-local variables and synchronization.
- [Call Stack](../../../concepts/call-stack.md) — The call stack size on CUDA devices of compute capability 2.x and higher can be queried and configured using specific runtime API functions, with overflow handling and compiler warnings managed through debugger integration and static analysis.
- [Texture and Surface Memory](../../../concepts/texture-surface-memory.md) — Texture and surface memory provide cached access to device memory, optimized for 2D spatial locality and offering performance benefits such as offloaded addressing calculations and data conversion.
- [Texture Memory](../../../concepts/texture-memory.md) — Texture memory is a read-only memory space in CUDA kernels accessed via texture fetches, configured through texture objects that define dimensionality, texel types, read modes, coordinate normalization, addressing modes, and filtering modes.
- [Texture Object API](../../../concepts/texture-object-api.md) — The Texture Object API provides a mechanism to create and manage texture objects in CUDA using resource and texture descriptions, enabling efficient texture sampling within kernels.
