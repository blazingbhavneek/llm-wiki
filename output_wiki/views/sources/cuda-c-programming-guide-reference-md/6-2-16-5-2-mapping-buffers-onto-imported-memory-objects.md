# 6.2.16.5.2 Mapping Bufers onto Imported Memory Objects

Part of [Cuda C Programming Guide Reference](README.md). Source lines L5934-L6297.

- [Mapping Buffers onto Imported Memory Objects](../../../concepts/mapping-buffers-imported-memory.md) — Describes how to map a device pointer onto an imported memory object (NvSciBufObj) using `cudaExternalMemoryGetMappedBuffer`, specifying offset and size, and notes that the resulting pointer must be freed using `cudaFree()`.
- [Mapping Mipmapped Arrays onto Imported Memory Objects](../../../concepts/mapping-mipmapped-arrays-imported-memory.md) — Describes how to map a CUDA mipmapped array onto an imported memory object using `cudaExternalMemoryGetMappedMipmappedArray` with parameters defined in `cudaExternalMemoryMipmappedArrayDesc`, noting that the number of mip levels must be 1.
- [Importing NvSciSync Objects](../../../concepts/importing-nvscisync-objects.md) — This page covers the process of creating NvSciSync attributes compatible with a CUDA device using `cudaDeviceGetNvSciSyncAttributes`, reconciling them, and importing the resulting NvSciSyncObj into CUDA as an external semaphore using `cudaImportExternalSemaphore`.
- [Signaling and Waiting on Imported Synchronization Objects](../../../concepts/signaling-waiting-imported-semaphores.md) — Describes how to signal and wait on imported NvSciSync-backed semaphore objects using `cudaSignalExternalSemaphoresAsync` and `cudaWaitExternalSemaphoresAsync`, including the use of flags to skip memory synchronization when cache coherency is not required.
- [CUDA Versioning and Compatibility](../../../concepts/cuda-versioning-compatibility.md) — Explains the distinction between Compute Capability and the CUDA Driver API version (CUDA_VERSION), detailing the backward-compatible but not forward-compatible nature of the driver API and the requirements for installed drivers relative to applications and libraries.
- [CUDA Compute Modes](../../../concepts/compute-modes.md) — CUDA compute modes (Default, Exclusive-process, and Prohibited) control device access granularity, while Compute Preemption on Pascal and later architectures allows instruction-level task switching to prevent monopolization.
- [Display Mode Switches and CUDA](../../../concepts/display-mode-switches.md) — Display mode switches alter the memory required for the GPU's primary surface, which can cannibalize CUDA memory allocations and cause subsequent CUDA runtime calls to fail with an invalid context error.
- [Tesla Compute Cluster (TCC) Mode](../../../concepts/tcc-mode.md) — TCC mode is a configuration for Tesla and Quadro Series devices on Windows that removes graphics functionality to optimize the device for compute tasks.
- [GPU Architecture Overview](../../../concepts/gpu-architecture-overview.md) — Overview of the NVIDIA GPU architecture, detailing the scalable array of Streaming Multiprocessors (SMs), SIMT execution model, hardware multithreading, and little-endian data representation.
- [SIMT Architecture and Warps](../../../concepts/simt-architecture.md) — SIMT (Single Instruction, Multiple Threads) architecture executes threads in groups of 32 called warps, where threads share an instruction stream but maintain independent state, with Volta architecture introducing Independent Thread Scheduling to allow per-thread concurrency and sub-warp divergence.
- [Hardware Multithreading and Resources](../../../concepts/hardware-multithreading.md) — Hardware multithreading in CUDA maintains execution contexts on-chip to eliminate switching costs, with resource limits and occupancy determined by register and shared memory partitioning per multiprocessor.
- [Performance Optimization Strategies](../../../concepts/performance-optimization-strategies.md) — Performance optimization in CUDA revolves around four basic strategies: maximizing parallel execution, optimizing memory and instruction usage, and minimizing memory thrashing, with efforts guided by profiling tools.
- [Maximizing Utilization](../../../concepts/maximizing-utilization.md) — Strategies to maximize GPU utilization by exposing parallelism across application, device, and multiprocessor levels through asynchronous execution, latency hiding via warp residency, and resource optimization.
- [Occupancy Calculator API](../../../concepts/occupancy-calculator.md) — The Occupancy Calculator API provides functions to predict and optimize GPU kernel execution configurations based on thread block size, cluster size, and shared memory usage.
