# 12.2.4. Cluster Launch Control for Thread Block Clusters

Part of [Cuda C Programming Guide Reference](README.md). Source lines L13543-L13857.

- [Cluster Launch Control](../../../concepts/cluster-launch-control.md) — Cluster Launch Control, introduced in Compute Capability 10.0, is a CUDA feature that enables thread blocks to cancel the launch of other pending thread blocks, facilitating work-stealing while maintaining reduced launch overheads and supporting preemption.
- [CUDA Dynamic Parallelism](../../../concepts/cuda-dynamic-parallelism.md) — An extension to the CUDA programming model enabling a CUDA kernel to create and synchronize with new work directly on the GPU.
- [Dynamic Parallelism Glossary](../../../concepts/dynamic-parallelism-glossary.md) — Definitions for key terms used in the Dynamic Parallelism section of the CUDA C Programming Guide, including Grid, Thread Block, Kernel Function, Host, Parent, Child, Thread Block Scope, and Device Runtime.
- [Dynamic Parallelism Execution Model](../../../concepts/dynamic-parallelism-execution-model.md) — The CUDA execution model extends to support dynamic parallelism by allowing device threads to configure, launch, and implicitly synchronize upon new grids, creating a nested parent-child hierarchy where the parent waits for all child grids to complete.
- [Dynamic Parallelism Synchronization](../../../concepts/dynamic-parallelism-synchronization.md) — Rules for synchronizing parent and child grids, including the deprecation of explicit `cudaDeviceSynchronize()` in device code and implicit synchronization upon grid exit.
- [Dynamic Parallelism Streams and Events](../../../concepts/dynamic-parallelism-streams-events.md) — Control of dependencies between grid launches using streams and events on the device, with strict scoping rules that define undefined behavior for cross-grid usage.
- [Dynamic Parallelism Ordering and Concurrency](../../../concepts/dynamic-parallelism-ordering-concurrency.md) — Kernel launch ordering follows CUDA Stream semantics, but no concurrency guarantees exist between different thread blocks or between parent and child grids.
- [Dynamic Parallelism Memory Model](../../../concepts/dynamic-parallelism-memory-model.md) — Parent and child grids share global and constant memory with weak consistency guarantees, while local and shared memory remain private and incoherent between execution contexts.
- [Dynamic Parallelism Programming Interface](../../../concepts/dynamic-parallelism-programming-interface.md) — The Device Runtime API provides a CUDA C++ interface for dynamic parallelism that mirrors the Host Runtime API, allowing per-thread kernel launches without inter-thread synchronization requirements.
- [Dynamic Parallelism Device-Side Kernel Launch](../../../concepts/dynamic-parallelism-device-kernel-launch.md) — Device-side kernel launch allows kernels to be launched from the device using the standard CUDA <<< >>> configuration syntax.
- [Dynamic Parallelism Launch Asynchronicity](../../../concepts/dynamic-parallelism-launch-asynchronicity.md) — Device-side kernel launches are asynchronous, allowing the launching thread to continue execution until an implicit synchronization point is reached.
- [Dynamic Parallelism Launch Environment Configuration](../../../concepts/dynamic-parallelism-launch-environment.md) — In dynamic parallelism, child kernels inherit global device configuration settings from the parent, while per-kernel configurations set from the host take precedence and cannot be overridden from the device.
