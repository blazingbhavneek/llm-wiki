# System Allocator

Part of [Cuda C Programming Guide Reference](README.md). Source lines L21681-L22199.

- [Hardware vs. Software Coherency in Unified Memory](../../../concepts/hardware-vs-software-coherency.md) — This page compares hardware coherent systems, which use combined page tables and cache-line granularity to minimize TLB misses and contention, with software coherent systems that rely on separate page tables and page-fault-based migration.
- [System Allocator (CUDA)](../../../concepts/system-allocator.md) — Demonstrates the use of system-allocated memory (malloc) in CUDA, highlighting the need for explicit memory advice (cudaMemAdvise) and the impact of hardware coherency (directManagedMemAccessFromHost) on CPU/GPU memory migration behavior.
- [Managed Memory (CUDA)](../../../concepts/managed-memory.md) — CUDA Managed Memory uses cudaMallocManaged to provide unified memory access, with behavior varying based on hardware coherency support (directManagedMemAccessFromHost) regarding page faults and data migration between CPU and GPU.
- [Host Native Atomics](../../../concepts/host-native-atomics.md) — Describes hardware-accelerated atomic accesses to CPU-resident memory on devices with hardware coherency, which eliminates the need for page fault emulation and is indicated by the cudaDevAttrHostNativeAtomicSupported attribute.
- [Unified Memory Atomics and Synchronization](../../../concepts/unified-memory-atomics-synchronization.md) — Covers CUDA Unified Memory support for atomic operations and heterogeneous synchronization primitives (cuda::atomic, cuda::barrier, etc.), detailing behavior on systems with and without CPU/GPU page tables.
- [Memcpy/Memset Behavior with Unified Memory](../../../concepts/memcpy-memset-unified-memory.md) — Explains how cudaMemcpy* and cudaMemset* functions interact with unified memory pointers, providing performance advice on using accurate cudaMemcpyKind hints, preferring cudaMemcpyDefault, avoiding initialization via these APIs, and using kernels or std::memcpy for system-allocated memory.
- [Unified Memory on Devices Without Full Support](../../../concepts/unified-memory-limited-support.md) — This section details the limitations and fallback mechanisms for Unified Memory on devices that do not fully support CUDA Unified Memory, specifically those limited to CUDA Managed Memory.
- [Managed Memory on Devices with Compute Capability 6.x+](../../../concepts/managed-memory-6-x.md) — CUDA Managed Memory is fully supported and coherent on devices with compute capability 6.x or higher without pageable memory access, though system allocators cannot be used.
- [Managed Memory on Windows or Compute Capability 5.x](../../../concepts/managed-memory-windows-5-x.md) — Describes CUDA Managed Memory v1.0 support on devices with compute capability lower than 6.0 or Windows platforms, highlighting limited support for data migration, coherency, and oversubscription.
- [Data Migration and Coherency (Pre-6.x)](../../../concepts/data-migration-coherency-pre-6-x.md) — Pre-6.x GPUs require all managed memory to be transferred to GPU memory before kernel launch to avoid faults, whereas 6.x+ introduces on-demand page faulting for seamless Unified Memory.
- [GPU Memory Oversubscription (Pre-6.x)](../../../concepts/gpu-memory-oversubscription-pre-6-x.md) — Devices with compute capability lower than 6.0 are restricted from allocating more managed memory than the physical size of the GPU memory.
- [Multi-GPU Managed Memory (Pre-6.x)](../../../concepts/multi-gpu-pre-6-x.md) — Details how managed allocations are visible to all GPUs via peer-to-peer on pre-6.x systems, describes migration to system memory if peer-to-peer support is missing, and outlines Windows-specific behaviors involving zero-copy fallback and environment variables.
- [Coherency and Concurrency (Pre-6.x)](../../../concepts/coherency-concurrency-pre-6-x.md) — On pre-6.x GPUs, simultaneous CPU and GPU access to managed memory is prohibited to guarantee coherency, meaning the CPU cannot access allocations while the GPU is active.
- [GPU Exclusive Access to Managed Memory (Pre-6.x)](../../../concepts/gpu-exclusive-access-pre-6-x.md) — On pre-6.x GPU architectures, the GPU holds exclusive access to managed memory while any kernel is active, requiring explicit CPU synchronization before any CPU access to managed data, even if the CPU and GPU are accessing different variables.
- [Explicit Synchronization and Logical GPU Activity](../../../concepts/explicit-synchronization-logical-activity.md) — Explicit synchronization is required to ensure GPU completion because Unified Memory relies on logical activity, where kernels are not guaranteed to have finished until a host synchronization call is issued.
- [Concurrent Access Rules and Examples](../../../concepts/concurrent-access-rules.md) — Defines the rules for concurrent access between CPU and GPU, and between multiple GPU kernels, when using managed and non-managed memory.
- [Managing Data Visibility with cudaStreamAttachMemAsync](../../../concepts/stream-attach-mem-async.md) — cudaStreamAttachMemAsync() allows finer-grained control over managed memory visibility by associating allocations with specific streams, enabling CPU access once that stream's operations complete even if other streams are active.
- [Stream Association Examples](../../../concepts/stream-association-examples.md) — Demonstrates the use of cudaStreamAttachMemAsync() to control CPU/GPU concurrency by associating managed memory with specific streams or the host, and highlights the consequences of improper association.
- [Stream Attach with Multithreaded Host Programs](../../../concepts/stream-attach-multithreaded.md) — Explains the primary use of cudaStreamAttachMemAsync() in multithreaded programs to enable independent task parallelism by associating CPU thread allocations with private streams.
- [Modular Programs and Data Access Constraints](../../../concepts/modular-programs-data-access.md) — Discusses the use of cudaMemAttachHost flag with cudaMallocManaged to create allocations initially invisible to device execution, ensuring safe independence between threads in modular programs, and mentions alternative barrier-based approaches.
- [Memcpy/Memset Behavior with Stream-Associated Unified Memory](../../../concepts/memcpy-memset-stream-associated.md) — Details the rules for cudaMemcpy* and cudaMemset* operations on devices where concurrentManagedAccess is not set, explaining how coherency in the copy stream determines whether data is accessed from host or device, and the implications for CPU access during these operations.
- [Lazy Loading (CUDA)](../../../concepts/lazy-loading.md) — Lazy Loading is a CUDA feature that delays the loading of modules and kernels until they are actually used, reducing initialization time and memory overhead.
- [Lazy Loading Version Support](../../../concepts/lazy-loading-version-support.md) — Lazy Loading requires CUDA 11.7+ Runtime and R515+ user-mode libraries, with no compiler support needed, and benefits are limited to libraries compiled with the newer toolkit.
- [Triggering Lazy Loading](../../../concepts/lazy-loading-triggering.md) — Explains how kernels and variables are loaded automatically upon first use and provides methods to explicitly trigger loading without execution: cuModuleGetFunction() for Driver API and cudaFuncGetAttributes() for Runtime API.
- [Querying Lazy Loading Status](../../../concepts/lazy-loading-query.md) — Describes how to check if Lazy Loading is enabled using cuModuleGetLoadingMode() after initializing CUDA.
- [Lazy Loading Issues: Concurrent Execution](../../../concepts/lazy-loading-issues-concurrent.md) — Lazy loading can cause deadlocks if context synchronization is required while kernels are executing concurrently, particularly in anti-patterns where one kernel waits for another.
- [Lazy Loading Issues: Allocators](../../../concepts/lazy-loading-issues-allocators.md) — Lazy loading delays kernel code loading to execution time, which can cause allocation failures if an application greedily reserves all VRAM at startup; solutions include using cudaMallocAsync(), adding buffer memory, or preloading kernels.
- [Lazy Loading Issues: Autotuning](../../../concepts/lazy-loading-issues-autotuning.md) — Lazy loading can skew benchmark results by including kernel load time, requiring warmup iterations or preloading to ensure accurate autotuning measurements.
