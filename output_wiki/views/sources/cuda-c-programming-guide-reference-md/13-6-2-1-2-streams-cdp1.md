# 13.6.2.1.2 Streams (CDP1)

Part of [Cuda C Programming Guide Reference](README.md). Source lines L14532-L15096.

- [CUDA Dynamic Parallelism (CDP1) Streams](../../../concepts/cuda-dynamic-parallelism-cdp1-streams.md) — Overview of stream handling in CUDA Dynamic Parallelism version 1 (CDP1), where streams are private to thread blocks, concurrency is not guaranteed, and explicit synchronization is deprecated.
- [CUDA Dynamic Parallelism (CDP1) Implicit (NULL) Stream](../../../concepts/cuda-dynamic-parallelism-cdp1-null-stream.md) — In CUDA Dynamic Parallelism version 1 (CDP1), the implicit NULL stream is shared between all threads in a block and does not insert implicit dependencies on pending work in other streams, unlike the host's NULL stream.
- [CUDA Dynamic Parallelism (CDP1) Events](../../../concepts/cuda-dynamic-parallelism-cdp1-events.md) — CUDA Dynamic Parallelism version 1 (CDP1) supports only inter-stream synchronization via `cudaStreamWaitEvent`, requiring events to be created with `cudaEventDisableTiming` and restricting handle scope to the creating thread block.
- [CUDA Dynamic Parallelism (CDP1) Synchronization](../../../concepts/cuda-dynamic-parallelism-cdp1-synchronization.md) — Details on `cudaDeviceSynchronize` in CDP1, which synchronizes on all work launched by any thread in the thread-block but does not imply intra-block synchronization; explicit synchronization is deprecated in CUDA 11.6 and removed for compute_90+.
- [CUDA Dynamic Parallelism (CDP1) Device Management](../../../concepts/cuda-dynamic-parallelism-cdp1-device-management.md) — Device management in CDP1 restricts control to the currently executing device, disabling cudaSetDevice and cudaGetDeviceProperties while supporting cudaGetDevice and cudaDeviceGetAttribute.
- [CUDA Dynamic Parallelism (CDP1) Memory Declarations](../../../concepts/cuda-dynamic-parallelism-cdp1-memory-declarations.md) — In CUDA Dynamic Parallelism version 1, file-scope `__device__` and `__constant__` variables behave identically to host launches, allowing all kernels to read/write device variables and share a uniform view of constant memory.
- [CUDA Dynamic Parallelism (CDP1) Textures and Surfaces](../../../concepts/cuda-dynamic-parallelism-cdp1-textures-surfaces.md) — In CUDA Dynamic Parallelism 1 (CDP1), dynamically created texture and surface objects can be passed to and used by child kernels, but their creation and destruction are restricted to the host, and legacy module-scope textures are not supported within device-launched kernels.
- [CUDA Dynamic Parallelism (CDP1) Shared Memory](../../../concepts/cuda-dynamic-parallelism-cdp1-shared-memory.md) — In CUDA Dynamic Parallelism version 1 (CDP1), shared memory can be declared statically or dynamically, but it cannot be passed directly to child kernels; data must be written back to global memory before launching children.
- [CUDA Dynamic Parallelism (CDP1) Symbol Addresses](../../../concepts/cuda-dynamic-parallelism-cdp1-symbol-addresses.md) — In CUDA Dynamic Parallelism version 1, device symbols are accessed directly via the address-of operator, making runtime symbol APIs redundant and enforcing read-only access for constant symbols.
- [CUDA Dynamic Parallelism (CDP1) API Errors](../../../concepts/cuda-dynamic-parallelism-cdp1-api-errors.md) — Error handling in CUDA Dynamic Parallelism (CDP1) involves per-thread error codes via cudaGetLastError() for launch failures and host-side reporting via cudaDeviceSynchronize() for device-side exceptions.
- [CUDA Dynamic Parallelism (CDP1) Launch Setup APIs](../../../concepts/cuda-dynamic-parallelism-cdp1-launch-setup.md) — Low-level launch APIs in CDP1. `cudaGetParameterBuffer` and `cudaLaunchDevice` are exposed via PTX and device runtime. These allow direct kernel launch from device code without the `<<<>>>` syntax.
- [CUDA Dynamic Parallelism (CDP1) API Reference](../../../concepts/cuda-dynamic-parallelism-cdp1-api-reference.md) — Overview of the subset of the CUDA Runtime API supported in device code for CUDA Dynamic Parallelism version 1, detailing synchronization, memory, and stream operations with specific semantic differences from host execution.
- [CUDA Dynamic Parallelism (CDP1) Device-side Launch from PTX](../../../concepts/cuda-dynamic-parallelism-cdp1-ptx-launch.md) — Low-level details for implementing dynamic parallelism in PTX using cudaLaunchDevice and cudaGetParameterBuffer.
- [CUDA Dynamic Parallelism (CDP1) cudaLaunchDevice](../../../concepts/cuda-dynamic-parallelism-cdp1-cdialaunchdevice.md) — Details on `cudaLaunchDevice` API in CDP1, including PTX and CUDA-level declarations, parameter buffers, and launch configuration.
- [CUDA Dynamic Parallelism (CDP1) cudaGetParameterBuffer](../../../concepts/cuda-dynamic-parallelism-cdp1-cdagetparameterbuffer.md) — Details on the cudaGetParameterBuffer API in CDP1, used to obtain a buffer for kernel parameters, including PTX and CUDA-level declarations and alignment requirements.
- [CUDA Dynamic Parallelism (CDP1) Parameter Buffer Layout](../../../concepts/cuda-dynamic-parallelism-cdp1-parameter-buffer-layout.md) — Rules for laying out parameters in the parameter buffer for CDP1, including alignment requirements, prohibition of reordering, and a 4KB size limit.
- [CUDA Dynamic Parallelism (CDP1) Toolkit Support](../../../concepts/cuda-dynamic-parallelism-cdp1-toolkit-support.md) — This page covers the first-generation toolkit support for CUDA Dynamic Parallelism, including instructions for including the Device Runtime API in CUDA code.
- [CUDA Dynamic Parallelism (CDP1) Including Device Runtime API](../../../concepts/cuda-dynamic-parallelism-cdp1-including-runtime.md) — Prototypes for the CUDA device runtime API are included automatically during program compilation, eliminating the need to explicitly include the cuda_device_runtime_api.h header.
- [CUDA Dynamic Parallelism (CDP1) Compiling and Linking](../../../concepts/cuda-dynamic-parallelism-cdp1-compiling-linking.md) — Compiling and linking CUDA programs with Dynamic Parallelism (CDP1) requires the `-rdc=true` flag and automatic or explicit linking against the static device runtime library `libcudadevrt`.
- [CUDA Dynamic Parallelism (CDP1) Programming Guidelines](../../../concepts/cuda-dynamic-parallelism-cdp1-programming-guidelines.md) — This page serves as a navigational reference for CUDA Dynamic Parallelism version 1 (CDP1) programming guidelines, directing users to the main documentation sections for basics and general guidelines.
- [CUDA Dynamic Parallelism (CDP1) Basics](../../../concepts/cuda-dynamic-parallelism-cdp1-basics.md) — CUDA Dynamic Parallelism (CDP1) allows device code to launch and synchronize with child kernels, utilizing a device runtime that is a functional subset of the host runtime.
- [CUDA Dynamic Parallelism (CDP1) Performance](../../../concepts/cuda-dynamic-parallelism-cdp1-performance.md) — Performance and synchronization considerations for CUDA Dynamic Parallelism version 1 are detailed in the CUDA C Programming Guide.
- [CUDA Dynamic Parallelism (CDP1) Synchronization Performance](../../../concepts/cuda-dynamic-parallelism-cdp1-synchronization-performance.md) — Explicit synchronization using cudaDeviceSynchronize() in device code is deprecated and inefficient; implicit synchronization at block end is preferred.
- [CUDA Dynamic Parallelism (CDP1) Kernel Overhead](../../../concepts/cuda-dynamic-parallelism-cdp1-kernel-overhead.md) — Overhead imposed by device runtime execution tracking when dynamic launches are active, potentially decreasing performance for library calls made from the device compared to the host side.
- [CUDA Dynamic Parallelism (CDP1) Implementation Restrictions](../../../concepts/cuda-dynamic-parallelism-cdp1-restrictions.md) — Hardware and software limitations in CDP1.
- [CUDA Dynamic Parallelism (CDP1) Runtime](../../../concepts/cuda-dynamic-parallelism-cdp1-runtime.md) — The CDP1 runtime section serves as a navigational pointer to the main Runtime documentation, directing users to the CDP2 version for detailed implementation specifics.
- [CUDA Dynamic Parallelism (CDP1) Memory Footprint](../../../concepts/cuda-dynamic-parallelism-cdp1-memory-footprint.md) — CUDA Dynamic Parallelism (CDP1) reserves device memory for parent-grid state synchronization and pending launch tracking, potentially requiring up to 860MB per parent generation.
- [CUDA Dynamic Parallelism (CDP1) Nesting and Synchronization Depth](../../../concepts/cuda-dynamic-parallelism-cdp1-nesting-depth.md) — CUDA Dynamic Parallelism (CDP1) limits nesting depth to 24 and sets a default synchronization depth of 2, which can be configured via cudaDeviceSetLimit but is deprecated in CUDA 11.6+.
- [CUDA Dynamic Parallelism (CDP1) Pending Kernel Launches](../../../concepts/cuda-dynamic-parallelism-cdp1-pending-launches.md) — Management of pending kernel launches in CUDA Dynamic Parallelism version 1 (CDP1) using a system-managed launch pool with fixed-size and virtualized components.
- [CUDA Dynamic Parallelism (CDP1) Configuration Options](../../../concepts/cuda-dynamic-parallelism-cdp1-configuration-options.md) — Configurable limits for CDP1 including sync depth, pending launch count, and stack size, managed via the cudaDeviceSetLimit() API.
- [CUDA Dynamic Parallelism (CDP1) Memory Allocation](../../../concepts/cuda-dynamic-parallelism-cdp1-memory-allocation.md) — In CUDA Dynamic Parallelism 1, device-side memory allocation uses a separate heap managed by `cudaLimitMallocHeapSize`, and cross-boundary free operations between host and device are prohibited.
- [CUDA Dynamic Parallelism (CDP1) SM Id and Warp Id](../../../concepts/cuda-dynamic-parallelism-cdp1-sm-warp-id.md) — In CUDA Dynamic Parallelism 1 (CDP1), the PTX registers %smid and %warpid are volatile, meaning their values may change if the device runtime reschedules thread blocks, making it unsafe to rely on them remaining constant.
- [CUDA Dynamic Parallelism (CDP1) ECC Errors](../../../concepts/cuda-dynamic-parallelism-cdp1-ecc-errors.md) — ECC error handling in CDP1 involves no in-kernel notification, with errors reported at the host side after the launch tree completes, potentially generating exceptions or continuing execution based on configuration.
