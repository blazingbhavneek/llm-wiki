# 24.1.2.4 Diference between Unified Memory and Mapped Memory

Part of [Cuda C Programming Guide Reference](README.md). Source lines L21101-L21680.

- [Difference between Unified Memory and Mapped Memory](../../../concepts/unified-memory-vs-mapped-memory.md) — CUDA Mapped Memory does not guarantee support for all memory operations like atomics on all systems, whereas Unified Memory does, though Mapped Memory is available on more systems.
- [CUDA Pointer Attributes](../../../concepts/cuda-pointer-attributes.md) — CUDA pointer attributes indicate how memory was allocated or registered rather than its physical location, allowing runtime detection of managed, host, device, or unregistered memory types via cudaPointerGetAttributes.
- [Runtime Detection of Unified Memory Support Level](../../../concepts/unified-memory-support-level-detection.md) — This page explains how to detect Unified Memory support levels at runtime using the CUDA API attributes cudaDevAttrPageableMemoryAccess and cudaDevAttrConcurrentManagedAccess.
- [GPU Memory Oversubscription with Unified Memory](../../../concepts/gpu-memory-oversubscription.md) — Unified Memory allows applications to allocate and share arrays larger than the memory capacity of any individual processor, enabling out-of-core processing without significant complexity.
- [Unified Memory Performance Hints](../../../concepts/unified-memory-performance-hints.md) — Performance hints for Unified Memory allow applications to provide CUDA with additional information to optimize decision-making, improving performance without impacting application semantics.
- [Data Prefetching in Unified Memory](../../../concepts/data-prefetching.md) — Details the cudaMemPrefetchAsync API for migrating data closer to a specified processor asynchronously, including examples for System Allocator and Managed Memory prefetching patterns.
- [Memory Discarding in Unified Memory](../../../concepts/memory-discarding.md) — The `cudaMemDiscardBatchAsync` and `cudaMemDiscardAndPrefetchBatchAsync` APIs allow applications to inform the CUDA runtime that memory contents are no longer useful, enabling the driver to skip redundant migrations and optimize performance.
- [Data Usage Hints (cudaMemAdvise)](../../../concepts/data-usage-hints.md) — Details the cudaMemAdvise API and its advice values: cudaMemAdviseSetReadMostly, cudaMemAdviseSetPreferredLocation, and cudaMemAdviseSetAccessedBy, including code examples and explanations of how these hints influence memory migration and access patterns.
- [Querying Data Usage Attributes on Managed Memory](../../../concepts/querying-data-usage-attributes.md) — Describes how to query memory range attributes set via cudaMemAdvise or cudaMemPrefetchAsync using the cudaMemRangeGetAttribute API, covering attributes for read-mostly hints, preferred locations, access history, and prefetch history.
- [Unified Memory on Devices with Full CUDA Unified Memory Support](../../../concepts/unified-memory-on-full-support-devices.md) — This section covers systems with full CUDA Unified Memory support, where the device can access any memory owned by the host process.
- [System-Allocated Memory: In-depth Examples](../../../concepts/system-allocated-memory-examples.md) — Demonstrates advanced usage patterns for system-allocated memory in CUDA Unified Memory, including host malloc, managed memory, stack variables, and global/extern variables, while highlighting the requirement to pass pointers for non-managed global variables.
- [File-backed Unified Memory](../../../concepts/file-backed-unified-memory.md) — File-backed Unified Memory allows GPU kernels to directly access memory mapped from host files, though atomic operations are unsupported on systems lacking hostNativeAtomicSupported.
- [Inter-Process Communication (IPC) with Unified Memory](../../../concepts/inter-process-communication-with-unified-memory.md) — Discusses IPC capabilities for System-Allocated Memory on full support systems, noting that CUDA IPC does not support Managed Memory and highlighting techniques like mmap, POSIX IPC, and memfd_create.
- [Unified Memory Performance Tuning](../../../concepts/unified-memory-performance-tuning.md) — General advice for achieving good performance with Unified Memory includes understanding paging, keeping data local, and tuning for memory transfer granularity, while noting that performance hints have associated costs.
- [Memory Paging and Page Sizes in Unified Memory](../../../concepts/memory-paging-and-page-sizes.md) — This page defines virtual and physical pages, page sizes, page tables, and TLBs, explaining how virtual addressing and page size choices impact performance in Unified Memory systems.
- [Choosing the Right Page Size](../../../concepts/choosing-page-size.md) — Details the trade-offs of small vs. large page sizes: small pages reduce fragmentation but increase TLB misses; large pages reduce TLB misses but increase migration latency and fragmentation, with a strong recommendation to tune virtual rather than physical page sizes.
- [Hardware vs. Software Coherency in Unified Memory](../../../concepts/hardware-vs-software-coherency.md) — This page compares hardware coherent systems, which use combined page tables and cache-line granularity to minimize TLB misses and contention, with software coherent systems that rely on separate page tables and page-fault-based migration.
- [Direct Unified Memory Access from Host](../../../concepts/direct-unified-memory-access-from-host.md) — Describes hardware support for coherent reads, stores, and atomic accesses from the host on GPU-resident unified memory, requiring specific device attributes and memory advice hints.
