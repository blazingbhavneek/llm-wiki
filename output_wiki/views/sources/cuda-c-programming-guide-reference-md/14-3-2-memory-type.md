# 14.3.2. Memory Type

Part of [Cuda C Programming Guide Reference](README.md). Source lines L15097-L15704.

- [Memory Type](../../../concepts/memory-type.md) — Introduction to CUDA 10.2+ memory type allocation via cuMemCreate and CUmemAllocationProp::allocFlags, allowing applications to specify memory features and ensuring device support.
- [Compressible Memory](../../../concepts/compressible-memory.md) — Compressible memory accelerates accesses to data with unstructured sparsity by saving DRAM and L2 bandwidth and capacity, allocated via CU_MEM_ALLOCATION_COMP_GENERIC and verified through device attribute queries.
- [Reserving a Virtual Address Range](../../../concepts/reserving-virtual-address-range.md) — Applications must reserve a virtual address range using cuMemAddressReserve to hold memory allocations; this range is initially unmapped, can be mapped to memory from any device, and must be returned using cuMemAddressFree after unmapping.
- [Virtual Aliasing Support](../../../concepts/virtual-aliasing-support.md) — Virtual Memory Management APIs allow creating multiple virtual mappings (proxies) to the same allocation via cuMemMap, where consistency between proxies is enforced using fence.proxy.alias PTX instructions or stream/event synchronization.
- [Mapping Memory](../../../concepts/mapping-memory.md) — The cuMemMap function associates physical memory allocations with reserved virtual address ranges, enabling users to map, unmap, and remap memory within the CUDA Virtual Memory Management API.
- [Controlling Access Rights](../../../concepts/controlling-access-rights.md) — Applications use the cuMemSetAccess function within the Virtual Memory Management APIs to explicitly grant or restrict device access to specific virtual address ranges, providing fine-grained control with minimal overhead compared to global peer access settings.
- [Fabric Memory](../../../concepts/fabric-memory.md) — Fabric Memory, introduced in CUDA 12.4, utilizes the CU_MEM_HANDLE_TYPE_FABRIC allocation handle to enable intra-node and inter-node memory sharing across Multi Node NVLINK systems, requiring device support verification and the NVIDIA IMEX daemon.
- [Multicast Support](../../../concepts/multicast-support.md) — Multicast Object Management APIs allow applications to leverage NVLINK SHARP for broadcast and reduction operations by creating Multicast Objects, forming Multicast Teams across NVSWITCH-connected GPUs, and using multimem PTX instructions.
- [Multicast Support Query](../../../concepts/multicast-support-query.md) — Applications must query device support for Multicast Objects using CU_DEVICE_ATTRIBUTE_MULTICAST_SUPPORTED before attempting to use them.
- [Allocating Multicast Objects](../../../concepts/allocating-multicast-objects.md) — Multicast objects are created using cuMulticastCreate with a CUmemAllocationProp structure that specifies the number of devices and handle types, ensuring the allocation size meets the granularity requirements obtained via cuMulticastGetGranularity.
- [Add Devices to Multicast Objects](../../../concepts/add-devices-to-multicast-objects.md) — Devices are added to a Multicast Team using cuMulticastAddDevice, a step that must be completed on all participating processes before binding memory to the Multicast Object.
- [Bind Memory to Multicast Objects](../../../concepts/bind-memory-to-multicast-objects.md) — Physical memory allocated with cuMemCreate is bound to a Multicast Handle using cuMulticastBindMem for each participating device.
- [Use Multicast Mappings](../../../concepts/use-multicast-mappings.md) — Multicast mappings enable collective memory operations in CUDA C++ using multimem PTX instructions, requiring compute capability 9.0 or higher.
- [Stream Ordered Memory Allocator](../../../concepts/stream-ordered-memory-allocator.md) — A legacy allocator (CUDA 11.3+) that enables ordering memory allocation/deallocation with stream work (cudaMallocAsync, cudaFreeAsync), improving memory reuse and reducing OS calls, but replaced by newer guides in CUDA 13.0.
- [Stream Ordered Memory Allocator Query](../../../concepts/stream-ordered-memory-allocator-query.md) — Support for the Stream Ordered Memory Allocator is queried via cudaDevAttrMemoryPoolsSupported, while IPC support is queried via cudaDevAttrMemoryPoolSupportedHandleTypes, requiring driver version checks to avoid errors on older drivers.
- [cudaMallocAsync and cudaFreeAsync](../../../concepts/cudamallocasync-and-cudafreeasync.md) — Core APIs for the Stream Ordered Memory Allocator that provide asynchronous memory management based on stream order, requiring explicit synchronization when sharing allocations across streams.
- [Memory Pools](../../../concepts/memory-pools.md) — Memory pools encapsulate virtual and physical memory resources, allowing allocations to be managed according to specific pool attributes and properties via APIs like cudaMallocAsync and cudaDeviceSetMempool.
- [Default Memory Pools](../../../concepts/default-memory-pools.md) — The default memory pool is a non-migratable, device-local allocation pool retrieved via cudaDeviceGetDefaultMempool, whose accessibility can be modified via cudaMemPoolSetAccess but which does not support IPC.
- [Explicit Memory Pools](../../../concepts/explicit-memory-pools.md) — Explicit memory pools allow applications to customize allocation properties such as IPC capability, maximum size, and NUMA node residency via the cudaMemPoolCreate API.
- [Physical Page Caching Behavior](../../../concepts/physical-page-caching.md) — The CUDA allocator minimizes OS calls by caching physical pages within a pool, controlled by the release threshold attribute, and allows explicit shrinking via cudaMemPoolTrimTo.
- [Resource Usage Statistics](../../../concepts/resource-usage-statistics.md) — Introduced in CUDA 11.3, resource usage statistics provide attributes to query current and high-watermark values for reserved and used memory within a CUDA memory pool.
- [Memory Reuse Policies](../../../concepts/memory-reuse-policies.md) — The CUDA stream ordered allocator attempts to reuse memory freed via cudaFreeAsync() before requesting new memory from the OS, with behavior controlled by specific pool attributes.
- [cudaMemPoolReuseFollowEventDependencies](../../../concepts/reuse-follow-event-dependencies.md) — This memory pool policy allows the allocator to reuse memory freed in one stream for allocations in another stream by examining dependencies established by CUDA events.
- [cudaMemPoolReuseAllowOpportunistic](../../../concepts/reuse-allow-opportunistic.md) — This policy enables the CUDA memory allocator to reuse freed memory based on stream progress semantics without requiring explicit event dependencies.
- [cudaMemPoolReuseAllowInternalDependencies](../../../concepts/reuse-allow-internal-dependencies.md) — This memory pool reuse policy allows the CUDA driver to insert dependencies into the allocating stream to reuse memory whose availability depends on another stream's progress when physical allocation fails.
- [Disabling Reuse Policies](../../../concepts/disabling-reuse-policies.md) — Users may disable controllable memory reuse policies to avoid run-to-run variance or non-deterministic serialization, requiring explicit synchronization on allocation failure.
- [Device Accessibility for Multi-GPU Support](../../../concepts/device-accessibility-multi-gpu.md) — Memory pool accessibility in CUDA is managed via cudaMemPoolSetAccess rather than peer access APIs, requiring devices to be peer-capable and applying settings globally to all pool allocations.
