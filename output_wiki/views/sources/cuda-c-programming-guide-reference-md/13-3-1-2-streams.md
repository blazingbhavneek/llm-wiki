# 13.3.1.2 Streams

Part of [Cuda C Programming Guide Reference](README.md). Source lines L13858-L14531.

- [CUDA Device Streams](../../../concepts/cuda-device-streams.md) — CUDA device streams allow for concurrent execution of kernels within a grid, requiring the use of non-blocking flags and tail launches for synchronization due to the lack of host-side synchronization APIs.
- [CUDA Implicit (NULL) Stream](../../../concepts/cuda-implicit-null-stream.md) — The CUDA device runtime provides a single implicit, unnamed stream shared by all threads in a block, which does not insert implicit dependencies on other streams due to the NonBlocking flag requirement.
- [CUDA Fire-and-Forget Stream](../../../concepts/cuda-fire-and-forget-stream.md) — The CUDA fire-and-forget named stream (cudaStreamFireAndForget) allows launching work without stream tracking overhead or dependencies on previous grids, but cannot be used for events and requires 64-bit compilation.
- [CUDA Tail Launch Stream](../../../concepts/cuda-tail-launch-stream.md) — The CUDA tail launch stream (cudaStreamTailLaunch) is a special stream that schedules a new grid for launch only after the parent grid and all its associated work have completed, acting as an implicit synchronization point.
- [CUDA Device Events](../../../concepts/cuda-device-events.md) — CUDA device runtime events provide limited support for inter-stream synchronization via cudaStreamWaitEvent, requiring the use of cudaEventDisableTiming as timing queries are unsupported.
- [CUDA Device Synchronization](../../../concepts/cuda-device-synchronization.md) — Explicit inter-thread synchronization from a parent thread to child grids is not supported, meaning visibility of changes in child grids to the parent grid cannot be guaranteed without explicit mechanisms like CUDA Events.
- [CUDA Device Management](../../../concepts/cuda-device-management.md) — Device runtime code is restricted to managing only the active GPU, with specific limitations on device switching and property querying APIs.
- [CUDA Device Constant and Device Memory](../../../concepts/cuda-device-constant-memory.md) — Explains that __device__ and __constant__ memory declared at file scope behave identically in device runtime, with all kernels having the same view of these variables.
- [CUDA Textures and Surfaces on Device](../../../concepts/cuda-textures-surfaces-device.md) — CUDA supports dynamically created texture and surface objects that can be created on the host and used on the device, but creation and destruction are restricted to the host side.
- [CUDA Shared Memory on Device](../../../concepts/cuda-shared-memory-device.md) — Shared memory in CUDA device runtime can be declared statically or dynamically via extern variables, but cannot be passed directly to child kernels launched recursively.
- [CUDA Symbol Addresses on Device](../../../concepts/cuda-symbol-addresses-device.md) — Describes how __device__ symbols can be referenced directly via the & operator within kernels, noting that CUDA runtime APIs for symbol management are unsupported by the device runtime and __constant__ data is read-only.
- [CUDA Device Errors and Launch Failures](../../../concepts/cuda-device-errors.md) — CUDA error handling relies on per-thread error codes retrieved via cudaGetLastError(), where device-side launch failures and exceptions (such as invalid address access) are propagated back to the host.
- [CUDA Launch Setup APIs](../../../concepts/cuda-launch-setup-apis.md) — Details the low-level device runtime APIs cudaGetParameterBuffer() and cudaLaunchDevice() used for kernel launches, including their C declarations, PTX mappings, and parameter buffer layout requirements.
- [CUDA Device API Reference](../../../concepts/cuda-device-api-reference.md) — This page lists the CUDA Runtime API functions supported in the device runtime, detailing specific behavioral differences from host execution such as per-thread error states and restricted stream/event flags.
- [PTX Device-Side Launch](../../../concepts/ptx-device-launch.md) — Low-level PTX support for dynamic parallelism kernel launches via specific APIs.
- [CUDA Toolkit Device Runtime Inclusion](../../../concepts/cuda-toolkit-device-runtime-inclusion.md) — Prototypes for the CUDA device runtime API are included automatically during program compilation, eliminating the need for explicit inclusion of the cuda_device_runtime_api.h header.
- [CUDA Toolkit Compilation and Linking](../../../concepts/cuda-toolkit-compilation-linking.md) — Details the compilation and linking process for dynamic parallelism, including the need to link against libcudadevrt and examples using nvcc.
- [CUDA Programming Guidelines Basics](../../../concepts/cuda-programming-guidelines-basics.md) — The CUDA device runtime provides a functional subset of the host runtime API, allowing device code to perform device management, kernel launching, and stream operations using syntax largely identical to the host API.
- [CUDA Performance Overhead](../../../concepts/cuda-performance-overhead.md) — Describes the potential performance overhead imposed by system software tracking dynamic launches, which is incurred when linking against the device runtime library.
- [CUDA Implementation Restrictions](../../../concepts/cuda-implementation-restrictions.md) — While Dynamic Parallelism guarantees all specified semantics, hardware and software resources are implementation-dependent and limit the scale, performance, and other properties of programs using the device runtime.
- [CUDA Runtime Memory Footprint](../../../concepts/cuda-runtime-memory-footprint.md) — The CUDA device runtime reserves memory for management tasks, specifically tracking pending grid launches, with configuration options available to reduce this footprint.
- [CUDA Pending Kernel Launches](../../../concepts/cuda-pending-launches.md) — Details the fixed-size launch pool that tracks configuration and parameter data until kernel completion, which is configurable via cudaDeviceSetLimit.
- [CUDA Configuration Options](../../../concepts/cuda-configuration-options.md) — CUDA configuration options allow host programs to control device runtime resource allocation, such as pending launch buffers and thread stack sizes, using the cudaDeviceSetLimit API before kernel execution.
- [CUDA Memory Allocation and Lifetime](../../../concepts/cuda-memory-allocation-lifetime.md) — Explains the distinct semantics of cudaMalloc/cudaFree between host and device, noting that device malloc maps to heap allocation and cross-environment free is an error.
- [CUDA SM ID and Warp ID](../../../concepts/cuda-sm-warp-id.md) — The PTX instructions %smid and %warpid provide volatile identifiers for the Streaming Multiprocessor and warp, respectively, but their values may change due to device runtime rescheduling, making them unsafe to rely on across the lifetime of a thread or thread block.
- [CUDA ECC Errors](../../../concepts/cuda-ecc-errors.md) — ECC errors are not reported to kernel code but are detected by the host after launch completion, potentially generating exceptions.
- [CDP2 vs CDP1](../../../concepts/cdp2-vs-cdp1.md) — This section summarizes the differences, compatibility, and interoperability between the new CDP2 and legacy CDP1 CUDA Dynamic Parallelism interfaces, including instructions for opting out of CDP2 on devices with compute capability less than 9.0.
- [CDP2 Differences](../../../concepts/cdp2-differences.md) — CDP2 removes explicit device-side synchronization and virtualized pools, introduces per-grid stream tracking and tail/fire-and-forget streams, and is restricted to 64-bit compilation.
- [CDP2 Compatibility and Interoperability](../../../concepts/cdp2-compatibility.md) — Explains compatibility rules, default behavior, and restrictions on mixing CDP1 and CDP2 functions in the same call graph.
- [Legacy CUDA Dynamic Parallelism (CDP1)](../../../concepts/cdp1-legacy.md) — This section serves as a reference for the legacy CDP1 implementation, directing users to the CDP2 documentation for current capabilities.
- [CDP1 Execution Environment](../../../concepts/cdp1-execution-environment.md) — The CDP1 execution environment and memory model details are documented in the main Execution Environment and Memory Model section, with specific CDP2 variations noted separately.
- [CDP1 Execution Environment Details](../../../concepts/cdp1-execution-environment-details.md) — Details the CUDA execution model primitives in the context of CDP1, including the deprecation warning for explicit synchronization.
- [CDP1 Parent and Child Grids](../../../concepts/cdp1-parent-child-grids.md) — CDP1 defines a properly nested execution model where a parent grid's completion is implicitly synchronized with the completion of all child grids it launches.
- [CDP1 Scope of CUDA Primitives](../../../concepts/cdp1-scope-primitives.md) — Describes the scope of streams and events in CDP1 (thread block scope) and the undefined behavior if used outside that scope.
- [CDP1 Synchronization](../../../concepts/cdp1-synchronization.md) — Details the deprecation of explicit synchronization with child kernels in CDP1 and explains the implicit synchronization behavior triggered when a parent block exits before its child launches complete.
- [CDP1 Streams and Events](../../../concepts/cdp1-streams-events.md) — In CDP1, device-created streams and events operate within thread block scope with undefined behavior for cross-block usage, while host-created streams and events are invalid within kernels.
- [CDP1 Ordering and Concurrency](../../../concepts/cdp1-ordering-concurrency.md) — Explains ordering semantics (in-order within stream) and the lack of concurrency guarantees between thread blocks in CDP1.
- [CDP1 Device Management](../../../concepts/cdp1-device-management.md) — The CDP1 device runtime lacks multi-GPU support and operates exclusively on the currently executing device, though it allows querying properties of all CUDA-capable devices.
- [CDP1 Memory Model](../../../concepts/cdp1-memory-model.md) — The CDP1 memory model directs users to the general memory model documentation, with specific details for the CDP2 version located elsewhere in the guide.
- [CDP1 Global Memory](../../../concepts/cdp1-global-memory.md) — Details coherence and consistency guarantees for global memory between parent and child grids in CDP1, specifying visibility points and synchronization requirements.
- [CDP1 Zero Copy Memory](../../../concepts/cdp1-zero-copy-memory.md) — Zero-copy memory shares the same coherence and consistency model as global memory and cannot be allocated or freed by device code.
- [CDP1 Constant Memory](../../../concepts/cdp1-constant-memory.md) — In CUDA Dynamic Parallelism version 1, constant memory is immutable from the device and is automatically inherited by child kernels from their parents.
- [CDP1 Shared and Local Memory](../../../concepts/cdp1-shared-local-memory.md) — In CUDA Dynamic Parallelism version 1 (CDP1), shared and local memory are private to thread blocks and threads respectively, are not visible or coherent between parent and child kernels, and referencing them outside their scope is undefined behavior.
- [CDP1 Local Memory](../../../concepts/cdp1-local-memory.md) — In CUDA Dynamic Parallelism version 1 (CDP1), local memory is private to a thread and cannot be passed as an argument to child kernels; such storage must instead be allocated from global memory.
- [CDP1 Texture Memory](../../../concepts/cdp1-texture-memory.md) — Details coherence rules for texture memory between parent and child grids in CDP1, noting that coherence is enforced at grid invocation and completion.
- [CDP1 Programming Interface](../../../concepts/cdp1-programming-interface.md) — The CDP1 programming interface is a specific version of the CUDA Dynamic Parallelism API, with detailed documentation referenced under the general Programming Interface section.
- [CDP1 CUDA C++ Reference](../../../concepts/cdp1-cpp-reference.md) — Describes the CDP1 language interface and API, noting its similarity to the host runtime and per-thread nature.
- [CDP1 Device-Side Kernel Launch](../../../concepts/cdp1-device-kernel-launch.md) — Details the <<<>>> syntax for launching kernels from the device in CDP1, including parameters for grid, block, shared memory, and stream.
- [CDP1 Launches are Asynchronous](../../../concepts/cdp1-launches-asynchronous.md) — In CUDA Dynamic Parallelism version 1 (CDP1), device-side kernel launches are asynchronous, allowing the parent thread to continue execution immediately without waiting for child kernels to complete.
- [CDP1 Launch Environment Configuration](../../../concepts/cdp1-launch-environment.md) — In CDP1, child kernels inherit global device configurations from their parent and cannot be reconfigured from the device.
