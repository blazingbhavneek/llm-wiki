# 6.2.8.7.5.3 Individual Node Update

Part of [Cuda C Programming Guide Reference](README.md). Source lines L2806-L3337.

- [Individual Node Update](../../../concepts/individual-node-update.md) — Covers the mechanism for updating parameters of instantiated graph nodes (cudaGraphExec_t) directly, avoiding the overhead of re-instantiation.
- [Individual Node Enable/Disable](../../../concepts/individual-node-enable.md) — The CUDA Graph API allows specific kernel, memset, and memcpy nodes in an instantiated graph to be enabled or disabled using cudaGraphNodeSetEnabled(), where disabled nodes are functionally empty and parameter updates take effect upon re-enablement.
- [Graph API Threading and Execution Constraints](../../../concepts/graph-api-threading.md) — This page details the threading safety rules for cudaGraph_t objects and the concurrency constraints for cudaGraphExec_t, clarifying that streams control ordering but not internal graph parallelism.
- [Device Graph Launch Overview](../../../concepts/device-graph-launch.md) — Device graph launch enables dynamic control flow on the GPU by allowing graphs to be launched from the device, requiring unified addressing and prohibiting concurrent device-side launches of the same graph.
- [Device Graph Creation](../../../concepts/device-graph-creation.md) — Device graph creation requires explicit instantiation using the cudaGraphInstantiateFlagDeviceLaunch flag, with structure fixed at instantiation time and performed exclusively on the host.
- [Device Graph Requirements](../../../concepts/device-graph-requirements.md) — This page details the constraints for launching graphs on a CUDA device, including single-device residency, allowed node types, restrictions on CUDA Dynamic Parallelism and MPS, and specific memory access rules for memcpy and memset operations.
- [Device Graph Upload](../../../concepts/device-graph-upload.md) — Device graph upload is the process of populating device resources before launching a CUDA graph, which can be done explicitly via cudaGraphUpload(), during instantiation via cudaGraphInstantiateWithParams(), or implicitly during the initial host launch.
- [Device Graph Update](../../../concepts/device-graph-update.md) — Covers the process of updating device graphs from the host and the requirement to re-upload the executable graph to the device for changes to take effect, noting undefined behavior if launched from the device during the update.
- [Device Launch Mechanics](../../../concepts/device-launch-mechanics.md) — Covers the mechanics of launching device graphs from host or device using cudaGraphLaunch(), noting that device graphs must be launched from another graph when launched from the device and that launch is per-thread.
- [Device Launch Streams and Modes](../../../concepts/device-launch-streams.md) — Device graphs cannot be launched into regular CUDA streams and must instead use distinct named streams that denote specific launch modes: Fire and Forget, Tail Launch, and Sibling Launch.
- [Fire and Forget Launch](../../../concepts/fire-and-forget-launch.md) — A fire-and-forget launch mode allows a graph to be submitted to the GPU immediately to run independently of the launching parent graph, with a limit of 120 such graphs per parent execution.
- [Execution Environments](../../../concepts/execution-environments.md) — Execution environments encapsulate device-side work for graphs and streams, providing a hierarchical model for synchronization where graph environments handle fire-and-forget work and stream environments parent graph launches.
- [Tail Launch](../../../concepts/tail-launch.md) — Tail Launch is a device-side graph execution mode that enables serial work dependencies by launching subsequent graphs only after a parent graph and all its children have completed.
- [Tail Self-Launch](../../../concepts/tail-self-launch.md) — Tail self-launch allows a device graph to enqueue itself for execution, facilitated by the `cudaGetCurrentGraphExec()` function which queries the handle of the currently running graph.
- [Sibling Launch](../../../concepts/sibling-launch.md) — Sibling launch is a variation of fire-and-forget launch where the graph is launched as a child of the launching graph's parent environment, ensuring it does not gate tail launches of the parent.
- [Conditional Graph Nodes](../../../concepts/conditional-graph-nodes.md) — Conditional graph nodes enable dynamic control flow, including IF, WHILE, and SWITCH logic, within CUDA graphs by evaluating device-set condition values to determine execution paths.
- [Conditional Handles](../../../concepts/conditional-handles.md) — Conditional handles (cudaGraphConditionalHandle) represent condition values in CUDA graphs, created via cudaGraphConditionalHandleCreate, associated with a single node, and optionally initialized with a default value at the start of each execution.
- [Conditional Node Body Graph Requirements](../../../concepts/conditional-node-body-requirements.md) — Defines the constraints for graphs used as conditional node bodies, including single-device residency, allowed node types, and restrictions on dynamic parallelism and memory access.
- [Conditional IF Nodes](../../../concepts/conditional-if-nodes.md) — Conditional IF nodes execute a body graph when a condition is non-zero, and optionally a second body graph when the condition is zero, introduced in CUDA 12.8.
- [Conditional WHILE Nodes](../../../concepts/conditional-while-nodes.md) — The WHILE conditional node type executes its body graph repeatedly as long as the condition remains non-zero, with evaluation occurring at execution start and after each body completion.
