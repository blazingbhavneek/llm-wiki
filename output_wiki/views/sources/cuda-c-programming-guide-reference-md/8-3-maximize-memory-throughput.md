# 8.3. Maximize Memory Throughput

Part of [Cuda C Programming Guide Reference](README.md). Source lines L6298-L6953.

- [Texture and Surface Memory](../../../concepts/texture-surface-memory.md) — Texture and surface memory provide cached access to device memory, optimized for 2D spatial locality and offering performance benefits such as offloaded addressing calculations and data conversion.
- [Occupancy-Based Kernel Launch](../../../concepts/occupancy-based-kernel-launch.md) — Demonstrates using the CUDA occupancy API (cudaOccupancyMaxPotentialBlockSize) to automatically determine optimal block size and grid size for a kernel launch based on device capabilities and kernel resource usage.
- [Cluster Occupancy API](../../../concepts/cluster-occupancy-api.md) — The Cluster Occupancy API provides functions and configuration structures to query maximum active clusters and potential cluster sizes, enabling the configuration of cluster-based kernel launches with specific dimensions.
- [CUDA Occupancy Header (cuda_occupancy.h)](../../../concepts/cuda-occupancy-h.md) — The cuda_occupancy.h header provides a standalone occupancy calculator and launch configurator implementation from Nsight Compute, useful for learning and environments where the full CUDA software stack is unavailable.
- [Maximize Memory Throughput](../../../concepts/maximize-memory-throughput.md) — Strategies for maximizing memory throughput include minimizing low-bandwidth host-device transfers, maximizing on-chip memory usage, and optimizing global memory access patterns.
- [Host-Device Data Transfer Optimization](../../../concepts/host-device-data-transfer.md) — Techniques to minimize and optimize data transfers between host and device, including batching, using page-locked memory, mapped memory, and handling integrated systems.
- [Device Memory Accesses](../../../concepts/device-memory-accesses.md) — Device memory accesses are handled via transactions that may be re-issued based on address distribution within a warp, with global memory throughput heavily dependent on coalescing and alignment.
- [Global Memory Alignment and Size Requirements](../../../concepts/global-memory-alignment.md) — Global memory accesses must use data types of 1, 2, 4, 8, or 16 bytes that are naturally aligned to compile into single instructions for optimal coalescing; misaligned accesses result in multiple instructions and potential data corruption.
- [Two-Dimensional Array Access Patterns](../../../concepts/2d-array-access-patterns.md) — Optimal access patterns for 2D arrays in global memory require width alignment to warp size, typically achieved using cudaMallocPitch or cuMemAllocPitch to add padding.
- [Local Memory](../../../concepts/local-memory.md) — Local memory is a device memory space used for automatic variables that cannot fit in registers, characterized by high latency and coalesced access patterns.
- [Shared Memory Banks and Conflicts](../../../concepts/shared-memory-banks.md) — Shared memory is divided into banks to allow simultaneous access for high bandwidth, but accessing multiple addresses within the same bank causes bank conflicts that serialize access and reduce throughput.
- [Constant Memory](../../../concepts/constant-memory.md) — Constant memory is a device memory space cached in the constant cache, where read requests are split based on distinct memory addresses, affecting throughput depending on cache hits.
- [Maximize Instruction Throughput](../../../concepts/maximize-instruction-throughput.md) — This section refers readers to the CUDA C++ Best Practices Guide for detailed strategies on optimizing instruction throughput.
- [Minimize Memory Thrashing](../../../concepts/minimize-memory-thrashing.md) — Strategies to avoid performance degradation from frequent memory allocations and frees, including sizing allocations appropriately, batching operations, and using cudaMallocManaged for oversubscription support.
- [Function Execution Space Specifiers](../../../concepts/function-execution-space-specifiers.md) — Function execution space specifiers (__global__, __device__, __host__) define where a function executes (host or device) and its callability, with __global__ and __device__ being mutually exclusive, while __host__ and __device__ can be combined.
- [Undefined Behavior in Cross-Execution Space Calls](../../../concepts/undefined-behavior-cross-execution.md) — Defines undefined behavior scenarios for calls between __host__ and __device__ functions based on __CUDA_ARCH__ definition.
- [Function Inline Specifiers](../../../concepts/function-inline-specifiers.md) — Details on __noinline__, __forceinline__, and __inline_hint__ qualifiers for controlling compiler inlining behavior.
- [Variable Memory Space Specifiers](../../../concepts/variable-memory-space-specifiers.md) — Overview of specifiers (__device__, __shared__, __constant__, etc.) that define the memory location of variables on the device.
- [Restricted Pointers (__restrict__)](../../../concepts/restricted-pointers.md) — The __restrict__ keyword allows programmers to assert that pointers do not alias, enabling compiler optimizations like common sub-expression elimination and code reordering, though it may increase register pressure and negatively impact occupancy in CUDA.
- [Built-in Vector Types](../../../concepts/built-in-vector-types.md) — CUDA provides built-in vector types derived from basic integer and floating-point types, allowing access to components via x, y, z, and w fields, along with specific constructor functions and alignment rules.
- [Built-in Variables](../../../concepts/built-in-variables.md) — Built-in variables provide information about grid and block dimensions and thread indices, and are valid only in device code.
- [Memory Fence Functions](../../../concepts/memory-fence-functions.md) — Memory fence functions in CUDA enforce sequentially-consistent ordering on memory accesses across different thread scopes to prevent data races in the weakly-ordered memory model.
