# 15.11. IPC Memory Pools

Part of [Cuda C Programming Guide Reference](README.md). Source lines L15705-L16131.

- [IPC Memory Pools](../../../concepts/ipc-memory-pools.md) — IPC capable memory pools provide an easy, efficient, and secure mechanism for sharing GPU memory between processes, utilizing a two-phase approach that first establishes security and then coordinates virtual address mappings.
- [cudaMemPoolExportToShareableHandle / cudaMemPoolImportFromShareableHandle](../../../entities/cuda-mem-pool-export-to-shareable-handle.md) — Sharing access to a memory pool involves retrieving an OS native handle via cudaMemPoolExportToShareableHandle(), transferring it using OS native IPC mechanisms, and creating an imported pool via cudaMemPoolImportFromShareableHandle().
- [cudaMemPoolSetAccess](../../../entities/cuda-mem-pool-set-access.md) — The cudaMemPoolSetAccess API enables access to imported memory pools from specific GPUs, as imported pools are initially only accessible from their resident device and do not inherit accessibility settings from the exporting process.
- [cudaMemPoolExportPointer / cudaMemPoolImportPointer](../../../entities/cuda-mem-pool-export-pointer.md) — These functions allow sharing memory pool allocations between processes via an opaque export data structure, requiring strict stream ordering and IPC event synchronization to ensure safe access and deallocation.
- [IPC Export Pool Limitations](../../../concepts/ipc-export-pool-limitations.md) — IPC pools currently do not support releasing physical blocks back to the OS, causing cudaMemPoolTrimTo to act as a no-op and cudaMemPoolAttrReleaseThreshold to be ignored.
- [IPC Import Pool Limitations](../../../concepts/ipc-import-pool-limitations.md) — Import pools in CUDA cannot be used for new allocations or trimming, and their resource usage statistics only reflect imported allocations.
- [Synchronization API Actions](../../../concepts/synchronization-api-actions.md) — Synchronization API actions optimize memory management by integrating the allocator with the CUDA driver, allowing the driver to guarantee completion of specific frees and release excess physical memory based on the release threshold during synchronization.
- [Addendums](../../../concepts/addendums.md) — The Addendums section of the CUDA C Programming Guide provides supplementary notes and specific behavioral details, such as those regarding the CUDA stream-ordered memory allocator.
- [cudaMemcpyAsync Current Context/Device Sensitivity](../../../concepts/cuda-memcpy-async-current-context-device-sensitivity.md) — cudaMemcpyAsync requires the calling thread's current context to match the stream's context when transferring memory allocated via cudaMallocAsync, whereas cudaMemcpyPeerAsync uses explicitly specified device primary contexts.
- [cuPointerGetAttribute Query](../../../concepts/cu-pointer-get-attribute-query.md) — Invoking cuPointerGetAttribute on an allocation after invoking cudaFreeAsync on it results in undefined behavior, regardless of stream accessibility.
- [cuGraphAddMemsetNode](../../../concepts/cu-graph-add-memset-node.md) — The cuGraphAddMemsetNode API does not support memory allocated via the stream ordered allocator, although memsets of such allocations can still be stream captured.
- [Pointer Attributes](../../../concepts/pointer-attributes.md) — The cuPointerGetAttributes function supports querying attributes for stream ordered allocations, including device ordinal and memory pool handle.
- [CPU Virtual Memory](../../../concepts/cpu-virtual-memory.md) — When using CUDA stream-ordered memory allocator APIs, avoid setting VRAM limitations with “ulimit -v” as this is not supported.
- [Graph Memory Nodes](../../../concepts/graph-memory-nodes.md) — Graph Memory Nodes are a component of the CUDA Graph API, documented in Chapter 16 of the CUDA C Programming Guide.
- [Graph Memory Nodes Introduction](../../../concepts/graph-memory-nodes-introduction.md) — Graph memory nodes enable graphs to create and own memory allocations with GPU-ordered lifetime semantics, allowing for fixed virtual addresses, driver-managed memory reuse, and physical memory aliasing across graph lifetimes.
- [Graph Memory Nodes Support and Compatibility](../../../concepts/graph-memory-nodes-support-compatibility.md) — Graph memory nodes require a CUDA driver version 11.4 or newer and GPU support for the stream ordered allocator, which can be verified using specific CUDA API calls.
- [Graph Memory Nodes API Fundamentals](../../../concepts/graph-memory-nodes-api-fundamentals.md) — Graph memory nodes represent allocation and free actions within CUDA graphs, where allocations are GPU-ordered and their lifetimes are tied to execution flow rather than API calls.
- [cudaGraphAddMemAllocNode / cudaGraphAddMemFreeNode](../../../entities/cuda-graph-add-mem-alloc-node.md) — Graph memory nodes allow explicit memory allocation and deallocation within a CUDA graph, requiring strict dependency ordering to ensure safe access to the allocated memory by kernel nodes.
- [Stream Capture Graph Memory](../../../concepts/stream-capture-graph-memory.md) — Graph memory nodes are created by capturing stream-ordered allocation and free calls (cudaMallocAsync and cudaFreeAsync), allowing the returned virtual addresses to be used by other operations within the graph while ensuring proper ordering via captured stream dependencies.
- [Accessing and Freeing Graph Memory Outside of the Allocating Graph](../../../concepts/accessing-freeing-graph-memory-outside-graph.md) — Graph memory allocations persist beyond the execution of the allocating graph and can be accessed by subsequent operations or other graphs, provided strict ordering via CUDA events, stream dependencies, or graph nodes is maintained before the memory is freed.
