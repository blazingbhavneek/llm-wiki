# 18.5.10.8 Operator Function

Part of [Cuda C Programming Guide Reference](README.md). Source lines L17267-L17983.

- [Operator Function](../../../concepts/operator-function.md) — An operator function in CUDA cannot be declared as a __global__ function.
- [Allocation and Deallocation Functions](../../../concepts/allocation-deallocation-functions.md) — User-defined operator new and delete functions cannot replace the built-in allocation and deallocation functions provided by the compiler for host or device code.
- [Classes in CUDA](../../../concepts/cuda-classes.md) — CUDA supports C++ classes with specific restrictions, notably prohibiting non-const static data members and preventing static member functions from being declared as __global__.
- [Virtual Functions in CUDA](../../../concepts/cuda-virtual-functions.md) — Virtual functions in CUDA require matching execution space specifiers for overrides, prohibit passing objects with virtual functions to global functions, and result in undefined behavior when invoking virtual functions across the execution space where the object was created.
- [Virtual Base Classes in CUDA](../../../concepts/cuda-virtual-base-classes.md) — CUDA prohibits passing objects of classes derived from virtual base classes as arguments to __global__ functions.
- [Anonymous Unions in CUDA](../../../concepts/cuda-anonymous-unions.md) — Member variables of a namespace scope anonymous union cannot be referenced in a __global__ or __device__ function.
- [Windows-Specific Class Layout Constraints](../../../concepts/cuda-windows-class-layout.md) — When using the Microsoft host compiler, CUDA follows the IA64 ABI for class layout while MSVC does not, causing layout mismatches for types with virtual functions, virtual bases, or multiple inheritance that result in undefined behavior when passed between host and device code.
- [Templates in CUDA](../../../concepts/cuda-templates.md) — CUDA restricts the use of certain types, such as those defined in host-only scopes, private class members, or unnamed types, as template arguments for __global__ function instantiations and __device__/__constant__ variable instantiations.
- [Trigraphs and Digraphs in CUDA](../../../concepts/cuda-trigraphs-digraphs.md) — CUDA does not support trigraphs on any platform and does not support digraphs on Windows.
- [Const-Qualified Variables in CUDA](../../../concepts/cuda-const-qualified-variables.md) — Namespace scope or static member const variables without execution space annotations are treated as host variables; their values can be used in device code if initialized with constant expressions and are of built-in floating point (except MSVC) or integral types, but references or addresses cannot be taken in device code.
- [Long Double in CUDA](../../../concepts/cuda-long-double.md) — The long double data type is not supported in CUDA device code.
- [Deprecation Annotation in CUDA](../../../concepts/cuda-deprecation-annotation.md) — The CUDA compiler (nvcc) supports standard and vendor-specific deprecation attributes for both host and device code, generating diagnostics during compilation that can be suppressed via specific flags or pragmas.
- [Noreturn Annotation in CUDA](../../../concepts/cuda-noreturn-annotation.md) — The CUDA compiler (nvcc) supports noreturn annotations for both host and device functions, accommodating various host compiler syntaxes and the C++11 standard attribute.
- [Likely/Unlikely Attributes in CUDA](../../../concepts/cuda-likely-unlikely-attributes.md) — The [[likely]] and [[unlikely]] standard attributes allow developers to hint to the CUDA device compiler optimizer about branch execution probability, though their use in host code without defined __CUDA_ARCH__ may trigger warnings on host compilers that do not support them.
- [Const and Pure GNU Attributes in CUDA](../../../concepts/cuda-const-pure-gnu-attributes.md) — The const and pure GNU attributes are supported for host and device functions to inform the optimizer about function side effects, with pure indicating no mutable state changes and const indicating no access or changes to mutable state.
- [__nv_pure__ Attribute in CUDA](../../../concepts/cuda-nv-pure-attribute.md) — The __nv_pure__ attribute is supported for both host and device functions, translating to platform-specific attributes on the host and signaling immutability to the device optimizer.
- [Intel Host Compiler Specifics in CUDA](../../../concepts/cuda-intel-host-compiler.md) — When using the Intel compiler as a host compiler, nvcc enables the __INTEL_COMPILER_USE_INTRINSIC_PROTOTYPES macro to support Intel intrinsic functions in host code.
- [C++11 Features in CUDA](../../../concepts/cuda-cpp11-features.md) — nvcc supports C++11 features enabled by the host compiler and can enable all C++11 features via the -std=c++11 flag, which configures the host toolchain accordingly.
- [Lambda Expressions in CUDA](../../../concepts/cuda-lambda-expressions.md) — Lambda expressions in CUDA derive their execution space specifiers from the enclosing function scope, and their closure types are restricted in usage within __global__ function template arguments.
- [std::initializer_list in CUDA](../../../concepts/cuda-initializer-list.md) — In CUDA, std::initializer_list member functions are __host__ __device__ by default, allowing device-side usage, but the --no-host-device-initializer-list flag restricts them to __host__ only.
- [Rvalue References in CUDA](../../../concepts/cuda-rvalue-references.md) — By default, std::move and std::forward are considered __host__ __device__. The --no-host-device-move-forward flag disables this, making them __host__ only.
- [Constexpr Functions in CUDA](../../../concepts/cuda-constexpr-functions.md) — By default, constexpr functions cannot be called from functions with incompatible execution spaces, but the --expt-relaxed-constexpr flag removes this restriction to allow cross-execution space calls.
- [Constexpr Variables in CUDA](../../../concepts/cuda-constexpr-variables.md) — Namespace scope or static member constexpr variables without execution space annotations are treated as host variables, allowing their values to be used in device code under specific scalar type constraints while prohibiting references or addresses.
- [Inline Namespaces in CUDA](../../../concepts/cuda-inline-namespaces.md) — The CUDA compiler injects references to device and global variables into host code, which can cause ambiguity errors if those variables are defined within inline namespaces that conflict with enclosing namespace definitions.
- [Inline Unnamed Namespaces in CUDA](../../../concepts/cuda-inline-unnamed-namespaces.md) — Entities such as __managed__, __device__, __shared__, __constant__ variables, __global__ functions, and surface/texture variables cannot be declared in namespace scope within an inline unnamed namespace.
- [Thread Local Storage in CUDA](../../../concepts/cuda-thread-local.md) — The thread_local storage specifier is not allowed in device code.
- [__global__ Functions in CUDA](../../../concepts/cuda-global-functions.md) — __global__ functions in CUDA are subject to specific restrictions regarding constexpr declarations, parameter types (excluding std::initializer_list, va_list, and rvalue references), variadic template pack placement, and the requirements for lambda closure types used as template arguments.
- [__managed__ and __shared__ Variables in CUDA](../../../concepts/cuda-managed-shared-variables.md) — __managed__ and __shared__ variables cannot be marked with the keyword constexpr.
- [Defaulted Functions in CUDA](../../../concepts/cuda-defaulted-functions.md) — In CUDA, execution space specifiers on non-virtual functions explicitly defaulted on their first declaration are ignored and inferred, whereas specifiers are preserved if the function is virtual or defaulted on a subsequent declaration.
- [C++14 Features in CUDA](../../../concepts/cuda-cpp14-features.md) — nvcc supports C++14 features enabled by the host compiler, activated via the -std=c++14 flag which configures the entire host toolchain to use the C++14 dialect.
- [Deduced Return Type in CUDA](../../../concepts/cuda-deduced-return-type.md) — __global__ functions cannot have deduced return types, and __device__ functions with deduced return types are converted to void by the frontend compiler, preventing their use in host code.
- [Variable Templates in CUDA](../../../concepts/cuda-variable-templates.md) — CUDA supports variable templates for device and constant variables, though the Microsoft host compiler imposes restrictions on const qualification.
